# AGENTS.md

## Project Purpose（本工程目的）

### Goal（目标）
构建一个锂电池数据仿真平台，为研发与策略迭代提供可复现、可比较的仿真结果。  
平台必须支持两类仿真：等效电路模型（ECM）与电化学模型（Electrochemical），并覆盖两类电池体系：三元锂电池与磷酸铁锂电池（LFP）。

### Non-Goals（非目标）
当前阶段不以生产级上车控制闭环为目标，不追求覆盖全部化学体系与全部老化机理。  
与仿真核心无直接关系的工作（如非必要 UI 装饰、无关集成）优先级低于仿真正确性与可复现性。

### Success Criteria（成功标准）
1. 标准流程可完成 ECM 与 Electrochemical 两类仿真。
2. 标准数据集可完成三元锂与 LFP 两类电池仿真。
3. 相同输入重复运行时，输出结构一致且关键结果可比较。
4. 任何任务若无法映射到上述目标，代理必须在“深度交互”中明确偏离风险并给出更优路径。

## Core Principles（核心原则）

1. 采用第一性原理思考（First-Principles Thinking），拒绝经验主义与路径盲从。
2. 代理不得假设用户完全清楚目标；当目标模糊时必须先澄清关键目标与验收标准。
3. 若目标清晰但路径非最优，代理必须主动提出更短、更低成本方案，并说明主要收益。

## Encoding Contract（编码规范）

1. 全项目文本文件统一使用 `UTF-8` 编码（允许 `UTF-8 BOM`）。
2. 新建或修改 `Markdown / Python / YAML / JSON / 日志` 文件时，必须显式使用 UTF-8，不得依赖系统默认编码。
3. 日志读写必须显式指定 `encoding="utf-8"`（如工具链强依赖可使用 `utf-8-sig`）。
4. 一旦发现乱码，必须先完成转码修复并确认可读，再继续后续编辑与提交。

## Response Contract（回复契约）

1. 复杂任务回复必须包含两个部分：
   - 直接执行：严格按用户当前要求交付可执行结果。
   - 深度交互：对原始需求进行审慎挑战（包括但不限于 XY 问题、路径弊端、替代方案）。
2. 简单任务可单段直答；但一旦出现目标不清、路径低效或潜在偏离，必须切回双段式结构。
3. 复杂任务的“直接执行”必须明确映射到 `Success Criteria` 中至少一条。
4. 复杂任务的“深度交互”必须检查是否存在目标偏移（XY 问题）并给出处理建议。
5. 对用户解释方案时，优先使用通俗表述，避免使用类似 `T0/T1/P0` 的阶段代号术语；如必须使用，需同步给出对应中文直白解释。

## Purpose Gate（目标对齐门槛）

1. 若用户请求与仿真核心目标不一致，代理必须先指出偏离，再给出低成本替代路径。
2. 若用户请求与目标一致但实现路径低效，代理必须给出更短路径（时间/复杂度/维护成本更低）。
3. 未通过 Purpose Gate 的方案不得直接推进为默认执行方案。

## Purpose Alignment Contract（公开行为契约）

- 输入（Input）：用户任务请求。
- 输出（Output）：在复杂任务中，代理必须提供任务与 `Goal/Success Criteria` 的映射说明，并在必要时给出偏离风险与替代路径。

## Deletion Safety Contract（删除安全契约）

1. 所有删除操作默认禁止直接永久删除（hard delete）。
2. 删除前必须先征得用户明确确认。
3. 删除动作必须移动到回收站，不得直接不可恢复删除。
4. 在 Git 工作区中，优先将本规则应用于仓库追踪文件（tracked files）。
5. 在非 Git 工作区中，默认保护工作区全部文件并执行同等确认流程。
